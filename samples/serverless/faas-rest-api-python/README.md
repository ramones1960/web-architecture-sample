# faas-rest-api-python

**カテゴリ**: serverless ・ **言語**: Python ・ **ステータス**: active

## 概要

サーバレス / FaaS（Function as a Service）で REST API を提供するアーキテクチャの最小サンプルです。
題材は最小限の **メモ（notes）API**。各エンドポイントを AWS Lambda + API Gateway の
プロキシ統合スタイルに倣った **純粋なハンドラ関数** `handler(event, context) -> response` として実装し、
HTTP サーバなしで個別にデプロイ・単体テストできることを示します。

クラウドに接続せずローカルだけで動かせるよう、API Gateway を模した
**ローカルインボーカ**（`local_invoker.py`）を同梱しています。
標準ライブラリのみで動作し、完全オフラインで実行できます。

## 役割（このアーキテクチャが担う役割）

APIサーバ（軽量・スパイクのある HTTP API）を、常駐サーバではなく
**クラウド関数（FaaS）** として提供する役割を担います。

- **リクエスト単位の課金・スケール**: 呼び出された分だけ実行され、その分だけ課金される。
  トラフィックの増減に応じて関数インスタンスが自動でスケールイン/アウトする。
- **運用（サーバ管理）の不要化**: OS パッチ、キャパシティ設計、常駐プロセスの監視といった
  運用負荷をクラウド側に寄せ、アプリのロジックに集中できる。
- **イベント駆動バックエンドにも適合**: HTTP（API Gateway）だけでなく、
  オブジェクトストレージへの保存・キュー・スケジュールなど多様なイベントを
  同じ「ハンドラ関数」モデルで受けられる。

## 採用基準（いつ採用する / 避ける）

| 採用すべきケース | 避けるべきケース |
| --- | --- |
| トラフィックが不定・スパイク的で予測しづらい | 常時高負荷で、関数の単価が常駐サーバより割高になる |
| 呼び出しが低〜中頻度 | コールドスタートが許容できないシビアな低レイテンシ要件 |
| 運用人員を割きたくない（サーバ管理を避けたい） | 長時間実行・重い処理（CPU/メモリ集約、バッチ的処理） |
| イベント駆動（オブジェクト保存トリガ、キュー消費など） | ベンダーロックインを避けたい（移植性を最優先したい） |
| 素早く小さく始めたい（PoC・新規プロダクト） | ステートフルで密なコネクションが必要（WebSocket 常時接続、大規模 DB コネクションプール） |

- **コールドスタート**: しばらく呼ばれていない関数は、初回呼び出し時にランタイム起動の
  オーバーヘッド（コールドスタート）が発生し、応答が遅くなることがある。
  低レイテンシが厳しい用途では注意が必要。
- **ステートレス前提**: 関数インスタンスはリクエスト間で状態を保持しない（破棄・複製され得る）。
  状態は必ず外部ストア（DynamoDB 等）に置く設計が前提となる。

## 構成

```text
   Client (curl / ブラウザ)
        |  HTTP
        v
+-------------------------------+
| API Gateway 相当               |
|  = local_invoker.py            |   HTTP <-> event/response の変換 + ルーティング
+-------------------------------+
        |  handler(event, context)
        v
+-------------------------------+
| handler 関数群 (src/handlers)  |   create_note / list_notes / get_note
|  純粋関数・HTTP を知らない      |   （Lambda の関数に相当）
+-------------------------------+
        |  add / list_all / get
        v
+-------------------------------+
| store (src/store.py)           |   インメモリ（= 外部DB相当のスタンドイン）
|  本番では DynamoDB 等に置換     |
+-------------------------------+
```

詳細は [docs/architecture.md](docs/architecture.md) を参照。

## 必要環境

- Python 3.11+（標準ライブラリのみ、外部依存なし）
- pip install 不要・ネットワーク不要・完全オフラインで動作

## 起動

```bash
python3 local_invoker.py
# => FaaS local invoker (API Gateway 相当) を起動しました: http://127.0.0.1:8000
# ポートは環境変数 PORT で変更可能（例: PORT=9000 python3 local_invoker.py）
```

別のターミナルから curl で各エンドポイントを呼び出せます。

```bash
# メモを作成（POST /notes）
curl -s -X POST http://127.0.0.1:8000/notes \
  -H 'Content-Type: application/json' \
  -d '{"text": "牛乳を買う"}'
# => {"id":"...","text":"牛乳を買う","createdAt":"..."}

# 一覧を取得（GET /notes）
curl -s http://127.0.0.1:8000/notes

# 1件取得（GET /notes/{id}） — 上で返った id を使う
curl -s http://127.0.0.1:8000/notes/<id>

# 存在しない id → 404
curl -s -o /dev/null -w '%{http_code}\n' http://127.0.0.1:8000/notes/none
# => 404
```

## テスト

```bash
python3 -m unittest discover -s tests
```

ハンドラには event dict を直接渡してテストします（HTTP サーバ不要）。
これが FaaS の純粋関数モデルがもたらすテスト容易性です。

## 設計上のトレードオフ

- ✅ 各エンドポイントが独立した純粋関数なので、HTTP なしで単体テストでき、移植性も高い。
- ✅ リクエスト単位の自動スケール・課金で、スパイク的トラフィックや低頻度用途と相性が良い。
- ✅ サーバ管理が不要で、小さく素早く始められる。
- ✅ ローカルインボーカにより、クラウド未接続でも同じハンドラを動かして再現できる。
- ⚠️ **コールドスタート**: 低頻度呼び出し時に初回応答が遅くなり得る。
- ⚠️ **ステートレス**: 状態は関数内に持てない。本サンプルのインメモリストアは
  ローカルデモ専用で、本番では外部ストア（DynamoDB 等）が必須。
- ⚠️ **ベンダーロックイン**: event/response の形やトリガはクラウド依存になりやすい
  （本サンプルはハンドラを純粋関数に保つことで影響を局所化している）。
- ⚠️ **ローカルでの再現性**: invoker はあくまで API Gateway の簡易模倣であり、
  認証・スロットリング・コールドスタート等の本番挙動までは再現しない。
