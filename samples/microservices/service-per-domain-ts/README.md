# service-per-domain-ts

**カテゴリ**: microservices ・ **言語**: TypeScript (Node) ・ **ステータス**: active

## 概要

小さな EC バックエンドを **ドメインごとに独立したサービス**へ分割し、その前段に
**API ゲートウェイ**を 1 枚置いた、マイクロサービス（service per domain）の最小サンプルです。

- **catalog-service**（4001）: 商品（id, name, priceYen）を管理。
- **order-service**（4002）: 注文を管理。注文作成時に catalog-service を **HTTP で同期呼び出し**して商品を検証する。
- **api-gateway**（4000）: 唯一の公開窓口。`/catalog/*` と `/orders/*` を各サービスへルーティングする。

各サービスは**独立した Node プロセス・独立したポート・独立したインメモリデータストア**を持ち、
コードやデータを共有しません。サービス間のやり取りは **HTTP のみ**です
（汎用の JSON ヘルパーだけは `shared/http.ts` で共有しますが、ドメインロジックは含みません）。
外部依存ゼロ・ビルド不要・完全オフラインで動作します。

## 役割（このアーキテクチャが担う役割）

**大規模業務システム（複数チームが独立して開発・デプロイ）のバックエンド**を担います。

- **API ゲートウェイ**がクライアントに対する唯一の公開窓口（single public surface）となり、
  ルーティング・転送を引き受ける。内部のサービス分割をクライアントから隠蔽する。
- **ドメイン別サービス**（catalog / order）が、それぞれの業務領域を**所有**する。
  各サービスは別チームが**独立して所有・デプロイ・スケール**できる単位であり、
  自分のデータストアだけを管理し、他サービスのデータには HTTP 経由でしかアクセスしない。

これにより「カタログチーム」と「注文チーム」が互いのリリースを待たずに開発・デプロイでき、
障害やスケール要件もサービス単位で分離できます。

## 採用基準（いつ採用する / 避ける）

| 採用すべきケース | 避けるべきケース |
| --- | --- |
| 大規模で、組織が複数チームに分かれている | 小規模・少人数（→ monolithic を推奨） |
| サービスごとに独立デプロイ／独立スケールが必要 | 強い整合性が必要な単一トランザクションが多い |
| 障害分離（ある領域の障害を他に波及させたくない） | 運用基盤（CI/CD・監視・分散トレーシング）が未整備 |
| 領域ごとに技術選定（言語・DB）を変えたい | ネットワーク越しの複雑さに見合う規模がない |

> **運用コスト / 分散システムの複雑さのトレードオフ**:
> マイクロサービスはプロセス境界をネットワーク境界へ置き換えます。その代償として、
> サービスディスカバリ・ネットワーク障害への対処・分散トレーシング・複数サービスの
> デプロイ/監視といった**運用コストが恒常的に発生**します。これらの運用基盤が整っておらず、
> 組織もまだ小さいうちは、**モノリスの方が総コストは低くなる**のが普通です。
> 「分割によって得られる独立性」が「分散システムの複雑さ」を上回る規模になって初めて採用を検討してください。

## 構成

```text
              ┌─────────────────────────────┐
   Client ──▶ │   API Gateway (4000)        │  ← 唯一の公開窓口（ルーティングのみ）
              └──────────────┬──────────────┘
                             │
            /catalog/*       │      /orders/*
        ┌────────────────────┴────────────────────┐
        ▼                                          ▼
┌──────────────────┐                     ┌──────────────────┐
│ catalog-service  │  ◀── 同期 HTTP 呼出 ──│  order-service   │
│   (4001)         │   GET /products/:id  │   (4002)         │
│  商品データ      │                     │  注文データ      │
│  (in-memory)     │                     │  (in-memory)     │
└──────────────────┘                     └──────────────────┘

※ order-service は POST /orders の処理中に catalog-service を同期呼び出しして
  商品の存在・価格を検証してから注文を確定する（サービス間通信）。
```

詳細は [docs/architecture.md](docs/architecture.md) を参照。

## 必要環境

- **Node.js 22+**（TypeScript を type stripping でビルドなしに直接実行します）。
- **外部依存なし**（npm install 不要）。`node:http` などの組み込みモジュールのみを使用。
- 完全オフラインで動作します。

## 起動

ワンコマンドで 3 サービスをまとめて起動（Ctrl+C で全停止）:

```bash
bash run.sh
```

個別に起動する場合（別ターミナルでそれぞれ実行）:

```bash
node services/catalog/server.ts   # catalog-service (4001)
node services/order/server.ts     # order-service   (4002)
node gateway/server.ts            # api-gateway      (4000)
```

ゲートウェイ（4000）に対する curl 例:

```bash
# 商品一覧（gateway -> catalog）
curl localhost:4000/catalog/products

# 単一商品
curl localhost:4000/catalog/products/p1

# 注文作成（gateway -> order -> catalog で検証 -> order が確定）
curl -X POST localhost:4000/orders -d '{"productId":"p1","qty":2}'

# 注文取得（作成時に返る id を使う。例: o1）
curl localhost:4000/orders/o1
```

## テスト

注文ドメインの純粋ロジック（合計計算・バリデーション・注文組み立て）の単体テスト。
他サービスの起動やネットワークは不要で、完全オフラインで実行できます。

```bash
node --test test/order.test.ts
```

## 設計上のトレードオフ

- ✅ サービスごとに**独立デプロイ・独立スケール**が可能。チームが並行して開発できる。
- ✅ **障害分離**: あるサービスの障害が他サービスへ波及しにくい（ゲートウェイで縮退も可能）。
- ✅ 領域ごとに**技術選定（言語・データストア）を変えられる**。
- ⚠️ **分散システムの複雑さ**: プロセス境界がネットワーク境界になり、設計・デバッグが難しくなる。
- ⚠️ **ネットワーク障害**: サービス間呼び出しは失敗しうる（本サンプルでは catalog 不達時に 502 を返す）。
  実運用ではタイムアウト・リトライ・サーキットブレーカ等が必要。
- ⚠️ **結果整合性**: データがサービスごとに分かれるため、サービスをまたいだ強い整合性は得にくい。
  複数サービスにまたがる更新は Saga 等のパターンで結果整合に倒すことが多い。
