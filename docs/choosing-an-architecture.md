# アーキテクチャの選び方 — 役割と採用基準

このドキュメントは、収録サンプルを **「どんな役割を担うか」** と **「いつ採用すべきか（採用基準）」** の
観点から横並びで比較し、最初の一歩の意思決定を助けるためのものです。

> 各サンプルの詳細な採用基準は、それぞれの `README.md` の
> 「役割」「採用基準（いつ採用する / 避ける）」セクションを正とします。
> 本ページは入口としての比較表です。

## 役割の早見表

| サンプル | カテゴリ | 言語 | 担う役割 | ひとことで言うと |
| --- | --- | --- | --- | --- |
| [layered-architecture-go](../samples/monolithic/layered-architecture-go/) | monolithic | Go | 業務アプリ（社内タスク管理）のバックエンド API | 1 つのデプロイ単位で素早く・保守しやすく作る |

## 採用基準の比較

| 観点 | layered (モノリス) | service-per-domain (マイクロサービス) | faas (サーバーレス) |
| --- | --- | --- | --- |
| **適した規模** | 小〜中規模 | 大規模 | 小〜中（イベント/スパイク向き） |
| **チーム構成** | 単一・少人数 | 複数チーム（サービス所有が分かれる） | 少人数でも可（運用負荷が小さい） |
| **デプロイ単位** | 1 つ（まとめてデプロイ） | サービスごとに独立 | 関数ごとに独立 |
| **スケール** | 全体を一律にスケール | サービス単位で個別スケール | リクエスト単位で自動スケール |
| **整合性** | 単一 DB トランザクションが容易 | 分散トランザクション/結果整合性が必要 | ステートレス前提・外部ストアに依存 |
| **運用負荷** | 低（1 つを運用） | 高（CI/CD・監視・分散トレーシングが前提） | 最小（インフラ運用が不要） |
| **主なコスト/弱点** | 大きくなると変更が衝突しやすい | ネットワーク越しの複雑さ | コールドスタート・ベンダーロックイン |

## 迷ったときの判断フロー

```text
まず monolithic（layered）から始めるべきか？
└─ Yes が基本。小さく速く立ち上げ、ドメインを学ぶ。

次のどれかに「強く」当てはまるか？
├─ 複数チームが同一コードで頻繁に衝突する / 領域ごとに独立デプロイ・独立スケールが要る
│     → microservices（service-per-domain）を検討
│       ※ ただし CI/CD・監視・分散トレーシングの運用基盤が整っていること
│
├─ トラフィックが不定・スパイク的 / サーバ運用に人を割きたくない / イベント駆動
│     → serverless（faas）を検討
│       ※ 常時高負荷・低レイテンシ必須・長時間処理には不向き
│
└─ いずれも強く当てはまらない
      → monolithic を維持し、必要になってから切り出す（早すぎる分割を避ける）
```

> 原則: **「最初からマイクロサービス」を避ける。** まずモノリスで境界を学び、
> 痛みが具体化してから（チームのスケール・独立デプロイの必要性が出てから）分割するのが安全です。
