# layered-architecture-go

**カテゴリ**: monolithic ・ **言語**: Go ・ **ステータス**: active

## 概要

レイヤードアーキテクチャ（4 層）で実装した、社内向けタスク管理（ToDo）バックエンドの HTTP API サンプルです。
依存方向を内側（ドメイン）へ向けることで、業務ロジックを技術的詳細（HTTP・永続化）から独立させています。
Go 標準ライブラリのみで動作し、外部依存・ネットワークアクセスなしで `go test ./...` が通ります。

## 役割（このアーキテクチャが担う役割）

このサンプルは **業務アプリ / 管理画面のバックエンド** という役割を想定しています。

- **単一デプロイ単位**: 1 つのバイナリをビルドして配置するだけで動く、運用がシンプルな構成。
- **中小規模**: 1 チームで把握・保守できる規模のドメインに向く。
- **CRUD 中心の業務ロジック**: タスクの作成・一覧・完了といった、データ操作を軸にした業務手続きを素直に表現する。

レイヤを明示的に分離することで、「画面まわりの都合（HTTP / JSON）」と「業務ルール」と「永続化技術」が混ざらず、
小さく始めつつも保守性を確保できます。

## 採用基準（いつ採用する / 避ける）

| 採用すべきケース | 避けるべきケース |
| --- | --- |
| 小〜中規模のシステム | 超大規模で、領域ごとの独立スケール / 独立デプロイが必須 |
| チームが小さく、単一コードベースで完結したい | 複数チームが同一コードベースで頻繁に衝突する |
| ドメインが比較的安定している | 領域ごとに技術スタック（言語・DB）を変えたい |
| まず素早く立ち上げたい | 部分的な独立リリースサイクルが必要 |
| レイヤ分離で保守性を確保したい | （→ これらの要件が強いなら **microservices** を検討） |

## 構成

依存方向は常に **内側（ドメイン）へ** 向きます。プレゼンテーション層とインフラ層が外側で、
アプリケーション層を経由してドメイン層に依存します。インフラ層はドメイン層が定義した
インターフェースを実装する（＝依存性逆転）ため、矢印はドメインへ向かいます。

```text
        ┌─────────────────────────────┐
        │        presentation         │  net/http ハンドラ（JSON 入出力・ステータス変換）
        └──────────────┬──────────────┘
                       │ depends on
                       ▼
        ┌─────────────────────────────┐
        │        application          │  TaskService（ユースケース: Create / List / Complete）
        └──────────────┬──────────────┘
                       │ depends on (interface)
                       ▼
        ┌─────────────────────────────┐
        │           domain            │  Task エンティティ・TaskRepository(interface)・ドメインエラー
        └──────────────▲──────────────┘
                       │ implements (依存性逆転)
        ┌──────────────┴──────────────┐
        │       infrastructure        │  MemoryTaskRepository（インメモリ実装・mutex で保護）
        └─────────────────────────────┘
```

詳細は [docs/architecture.md](docs/architecture.md) を参照してください。

## 必要環境

- Go 1.24+（標準ライブラリのみ、外部依存なし）

## 起動

```bash
go run ./cmd/server
# PORT 環境変数でポート変更可（既定は 8080）
PORT=9090 go run ./cmd/server
```

各エンドポイントの呼び出し例（別ターミナルで実行）:

```bash
# タスク作成
curl -s -X POST http://localhost:8080/tasks \
  -H 'Content-Type: application/json' \
  -d '{"title":"請求書を確認する"}'
# => 201 Created {"id":"...","title":"請求書を確認する","done":false,"createdAt":"..."}

# タスク一覧
curl -s http://localhost:8080/tasks
# => 200 OK [{"id":"...","title":"請求書を確認する","done":false,"createdAt":"..."}]

# タスク完了（上記で得た id を指定）
curl -s -X POST http://localhost:8080/tasks/<id>/complete
# => 200 OK {"id":"...","done":true,...}

# 不正な入力（空タイトル）
curl -s -X POST http://localhost:8080/tasks \
  -H 'Content-Type: application/json' -d '{"title":""}'
# => 400 Bad Request {"error":"task: title must not be empty"}
```

| メソッド | パス | 説明 | 主なステータス |
| --- | --- | --- | --- |
| GET | `/tasks` | タスク一覧を取得 | 200 |
| POST | `/tasks` | タスクを作成 | 201 / 400 |
| POST | `/tasks/{id}/complete` | タスクを完了にする | 200 / 404 |

## 設計上のトレードオフ

- ✅ **関心の分離**: HTTP・業務ルール・永続化が層ごとに分かれ、各層を独立して理解・変更できる。
- ✅ **テスト容易性**: ドメインがインターフェースに依存するため、インメモリ実装や時刻・ID 注入で高速かつ決定的にテストできる。
- ✅ **差し替え可能性**: `TaskRepository` の実装を RDB 版に差し替えても上位層のコードは変わらない。
- ✅ **運用のシンプルさ**: 単一バイナリ・単一デプロイ単位で、立ち上げが速い。
- ⚠️ **層をまたぐ単純 CRUD の冗長さ**: 小さな機能でも複数層を通すため、ボイラープレートが増えやすい。
- ⚠️ **スケールの限界**: 領域ごとの独立スケール / 独立デプロイには向かない（必要なら microservices を検討）。
- ⚠️ **層の境界の維持コスト**: 規律がないと上位層がインフラ詳細に依存し、層が形骸化しやすい。
